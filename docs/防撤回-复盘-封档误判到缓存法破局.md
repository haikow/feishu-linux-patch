# Linux 飞书防撤回 复盘：为什么"封档"是误判，缓存法怎么破的

> 目标：Linux 桌面飞书（Electron，`/opt/bytedance/feishu`）让被撤回的消息原文继续显示。
> 结论：**能做**，但不是"把原文渲染出来"，而是**撤回前就把内容缓存下来、撤回时取回拼上**。
> 参考：本目录同源的 Windows 吾乐吧补丁（`FeiShuRevokeMsgPatcher`）+ 安卓 SQL 层缓存原理。

---

## 一、为什么当初判了"封档"（错在哪）

早期结论：「JS/asar 层做不到，撤回从 native 进来改数据、UI 被动显示，封档」。这个判断错在一个**隐含前提**——以为防撤回 = **在撤回渲染时把原文渲染出来**。围着这个前提试了 6 轮：

- 把 `recallMessage` / `groupOwnerRecallMessage` store 方法置空；
- `isRecalled` 渲染门控置 `!1`；
- `recallType!==NOT_RECALL?撤回span:内容` 置死；
- 主组件 `renderRecalledMessage()` 门控置 `if(!1)`。

全部无效。运行时诊断（往 `renderMessageContent` 注入 localStorage dump）还发现"代码根本不执行"，于是误判"真正的撤回渲染模块定位不到 / 在 native 之下"。

**真正的原因**：
1. **桌面撤回会把原文抹掉**——门控就算置死，落到 `getByContext` 也没内容可渲染（和安卓 SQL `REPLACE` 抹内容同理）。
2. 撤回实际走的是**"xxx撤回了一条消息"系统提示**渲染（`system-text`），**不是**气泡占位 `renderRecalledMessage()`。当年盯错了渲染路径，所以探针"不执行"。

> 教训：**"把被删的东西渲染出来"这个前提本身就错了**——数据都没了，渲染层无米下炊。正确姿势是**在数据还在的时候先存一份**。

## 二、破局：抄 Windows 补丁的缓存法

当前目录的 Windows 补丁（`FeiShuRevokeMsgPatcher.exe` + `Plugins/*.dll`）就是干这个的。它的注入 JS 是 **.NET UTF-16 字符串**，普通 `strings` 抓不到，要 `strings -e l`（16 位小端）：

```bash
strings -e l -n 8 Plugins/FeiShuRevokeMsgPatcher.Plugin.Install.dll | grep -iE 'recalledMessage|upsertPreviews|updateMessagesMeRead'
```

抠出来看懂它的两段逻辑：
1. **`upsertPreviews`**（会话预览更新，撤回**前**就发生）：把预览的 `{id, 内容}` 缓存进 `localStorage.recalledMessageCacheList`；
2. **撤回系统提示渲染**：按消息 id 从缓存取回原文，拼到"撤回了一条消息"后面。

（顺带发现它的"永不已读" = `if(...){t.messageIds=[]}` + `updateMessagesMeRead`，正是我们 Linux 已做的防已读。）

## 三、移植到 Linux（`patch_feishu.py` 的 3 个注入点）

Linux 版 asar 结构和 Windows 不同（新版用 `i.jsx` JSX runtime、类方法而非内联 IIFE），但**飞书服务端数据模型一致**，预览对象字段相同，逻辑可直接搬：

- **3a 缓存写** → `b3a134417f.js` 的 `upsertPreviews:(e,t)=>{...forEach(t=>{` 里注入：
  `t.localizedDigestMessage` 存在时，取 `t.digest.elements` 转 richText、配 `t.lastVisibleMessageId`/`t.lastMessageType`，去重存进 `localStorage`（上限 3000）。
- **3b 缓存读** → `ab3699555f.js` 的撤回提示 `content:__Text("ews",{name:"<name />"})` 后拼 `+window.__flRecallText(E.id)`（`E`=消息变量，正则捕获，跨版本稳）。
- **3c 取回函数** → `window.__flRecallText(id)` 注入撤回提示文件头，读缓存按 id 取 `richText.elements` 拼文本，返回 `": 原文"`。

全程 `try/catch` 兜底：字段猜错只是不显示、绝不崩。效果实测：撤回显示「**xxx撤回了一条消息: 原文内容**」✅（2026-07-12）。

**边界**（同 Windows）：缓存来自会话预览=各聊天**最后一条**，撤回最近消息最稳；很老的历史消息可能没缓存；只对打补丁后新到的消息有效；图片/表情显示 `[图片]`/`[表情]`。

---

## 四、方法论沉淀

1. **"渲染层复原"对"删除型"功能是死路**——数据被抹了就没得渲染。防撤回/防删除的正解是**提前缓存**（安卓在 SQL 写入前留副本、桌面在预览到达时存 localStorage），不是事后渲染。
2. **"代码不执行"要先怀疑盯错了路径**，而不是急着下"不可达"结论。撤回有多条渲染路径（气泡占位 vs 系统提示），盯错一条会得出全盘错误结论。
3. **同源跨平台补丁是金矿**。Windows 吾乐吧补丁和 Linux 是同一套 Electron 代码，它的注入 JS 直接告诉你"存哪、取哪"。`.NET 字符串是 UTF-16，用 `strings -e l` 抠**。
4. **封档结论要标前提**。"JS 层做不到"只在"事后渲染"前提下成立；换成"提前缓存"前提，立刻可做。和安卓防已读那次一样——**前提一换，结论就翻**。

*相关：安卓端另有实现（hook SQLite 存储层的 `sqlite3_step`，撤回写库时 id+cid 双改绑，原文行不被覆盖）——与桌面缓存法思路不同、殊途同归。*
